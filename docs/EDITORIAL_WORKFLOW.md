# Editorial Workflow

This document turns the principles in [the book blueprint](BOOK_BLUEPRINT.md) and [quality rubric](QUALITY_RUBRIC.md) into the repository's canonical operational workflow. `AGENTS.md` remains the concise project entrypoint. [The chapter lifecycle](CHAPTER_LIFECYCLE.md) owns state definitions and evidence requirements.

## Canonical invocation

From the repository root, invoke:

```text
$ebook Continue the book autonomously until the next human decision is required.
```

The workflow discovers the target and phase from repository evidence. A chapter number, title, or status is not required. Topic, re-review, and stop-after-phase requests are optional overrides; they cannot bypass prerequisites, review, validation, or human approval.

## Repository-state discovery

Every invocation begins before content generation with this sequence:

1. read `book-manifest.json` and the applicable repository guidance;
2. inspect chapter, review, glossary, reference, status, and generated-book artifacts;
3. run `python scripts/validate_book.py`;
4. run `python scripts/next_book_action.py`;
5. construct an evidence model for every chapter;
6. compare each recorded state with the most advanced state justified by evidence;
7. repair only unambiguous metadata drift or stop on an ambiguous conflict;
8. inspect the selected chapter's detailed evidence before acting.

The deterministic next-action command recommends work but neither generates prose nor changes repository state. The main agent remains responsible for checking the relevant evidence and repository guidance.

## Sources and synchronization

Each artifact has one responsibility:

- `docs/BOOK_BLUEPRINT.md` owns planned intellectual architecture: parts, chapter purpose, core questions, and suggested initial writing order.
- `book-manifest.json` owns stable chapter IDs, numbering, workflow state/history, expected paths, required/advisory dependencies, priorities, review score index, update date, and human-gate state.
- `docs/BOOK_STATUS.md` is generated by `python scripts/sync_book_status.py`; it is the human-readable projection of the manifest, not a competing state store.
- `chapters/*.md` own manuscript content and chapter metadata.
- `reviews/*.md` own architecture, research, findings, revision history, reviewer re-checks, rubric evidence, validation records, candidate packages, and human decisions.
- `book.md` is generated integrated output. It includes only chapters whose manifest state is `Integrated`.

After a manifest mutation, regenerate the status projection. After an integration transition, regenerate the book. Run full validation after either operation. Do not edit generated chapter rows in `BOOK_STATUS.md` or prose in `book.md` directly.

## Work-selection priority

Use these categories in strict order.

### 1. Repository inconsistency

Repair or report inconsistent state before generating content. Examples include manifest/status drift, a recorded state without required evidence, missing chapter/review relationships, invalid approval records, broken links, an invalid generated build, or illegal lifecycle history.

Automatically repair metadata only when actual repository evidence makes one correction uniquely valid. Never manufacture substantive evidence or silently choose between plausible states.

### 2. Human gate

If any chapter is `Candidate for Human Approval` or any book-level editorial question requires a human decision, prepare one concise approval package, identify the exact decision, and stop. Do not begin unrelated work.

### 3. Active incomplete work

Resume the active chapter furthest through a justified lifecycle. Start at its first incomplete phase. Do not repeat completed phases unless validation invalidates them, material revisions require an affected re-review, or evidence is insufficient.

### 4. Approved but unintegrated work

Verify approval evidence, validate, transition to `Integrated`, synchronize status, rebuild the generated book, update necessary cross-references without altering approved prose, and run full validation.

### 5. New chapter selection

Begin a new chapter only when the earlier categories are empty. Select from chapters whose required dependencies are `Integrated`. Rank by:

1. explicit editorial priority;
2. recommended writing priority;
3. satisfied dependencies;
4. learning-sequence coherence;
5. foundational value to later chapters;
6. cross-book integration needs.

Chapter number is a reading/build order, not an orchestration rule. Record selection reasoning in the new review file.

## Chapter execution

Once state discovery selects a target and phase, continue along this evidence-backed path:

```text
Plan architecture if missing
        ↓
Evaluate architecture
        ↓
Research necessary claims
        ↓
Draft
        ↓
Run independent reviewers
        ↓
Aggregate findings
        ↓
Revise blocking and major findings
        ↓
Rerun affected reviewers
        ↓
Deterministic validation and rubric assessment
        ↓
Candidate for Human Approval
```

Use at most three major revision cycles. After the third unsuccessful cycle, keep `Revision Required`, record unresolved issues, and stop with a focused blocker report. Do not lower the rubric threshold.

## Architecture pass

Before drafting, the review must record purpose, prerequisites, learning outcomes, core mental model, scope boundaries, deferred concepts, example strategy, cross-chapter relationships, research questions, risks, and the reason this chapter is the valid target. Evaluate the plan against the blueprint, rubric, prerequisite availability, duplication risk, and dependency boundaries.

An optional request to stop after architecture planning stops only after the plan and evaluation have been recorded and the state is `Architecture Ready`.

## Draft and research pass

Draft from the evaluated architecture. Teach from first principles, preserve language independence, use realistic examples, qualify context-dependent behavior, explain failure modes and trade-offs, and connect decisions to production, product, and business outcomes.

Research only claims that need evidence. Prefer standards, RFCs, official documentation, original papers, and authoritative project documentation. Record used sources in `references.md`; do not turn the chapter into a literature review or fabricate support.

## Independent review pass

Run these project-scoped agents in bounded parallel after a complete draft:

- `technical-reviewer`: correctness, mechanisms, assumptions, and trade-offs;
- `pedagogy-reviewer`: first-principles progression, transfer, exercises, and mastery;
- `production-reviewer`: failure, operability, user impact, and business consequences;
- `continuity-editor`: terminology, dependencies, duplication, links, and cross-book integration;
- `source-verifier`: primary-source support, citation fit, qualification, and uncertainty.

All reviewers use `sandbox_mode = "read-only"`, return findings only, and cannot delegate because `.codex/config.toml` sets `agents.max_depth = 1`. Before spawning, verify that the active runtime has not weakened the read-only agent configuration; stop if that guarantee is unavailable. The main agent is the only writer and records findings in the review file. It aggregates independent results only after all reviewers return.

Findings use `Blocking`, `Major`, `Minor`, and `Optional`. Every blocking or major finding includes a location, consequence, concrete revision direction, and affected rubric criteria. Materially conflicting reviewer conclusions that repository guidance cannot resolve create a human gate.

## Revision and re-check pass

Resolve every blocking and major finding that can be settled from repository guidance. Record the action and evidence. Rerun reviewers whose lens is materially affected; a wording-only change does not require all five reviewers, while conceptual restructuring usually does.

Routine wording, local structure, example choice, and nonstrategic minor findings do not require human input. Cross-book architecture changes, source uncertainty, contradictions, or changes that may invalidate approved chapters do.

## Candidate assessment

Before becoming a candidate, the chapter must have:

- complete architecture, draft, reviewer, and revision evidence;
- zero unresolved blocking and major findings;
- a passed `python scripts/validate_book.py` result;
- evidence-backed scores for all twelve rubric criteria;
- an average of at least 4.0/5 and no criterion below 3/5;
- resolved material accuracy concerns and placeholders;
- required glossary, references, and cross-links;
- a concise approval package explaining scope, strengths, remaining non-blocking limitations, score, and the exact decision requested.

The main agent transitions the chapter to `Candidate for Human Approval`, records a pending gate, synchronizes status, and stops.

## Human approval

Only a human may make this transition:

```text
Candidate for Human Approval
        ↓
Approved
```

The review evidence must record `decision = approved`, `actor = human`, and the date; the manifest history must record the same human actor. An agent may prepare the package or record a human decision that was actually given, but it may not decide or infer approval.

Stop for human input when a candidate is ready, cross-book architecture would change, requirements conflict, material reviewers disagree, approved chapters could be invalidated, or source evidence remains materially uncertain.

## Integration

Integration is mechanical editorial publication, not a prose-revision phase:

1. verify explicit human approval evidence;
2. run deterministic validation;
3. append the `Integrated` manifest transition;
4. update any required cross-references without changing approved chapter meaning;
5. run `python scripts/sync_book_status.py`;
6. run `python scripts/build_book.py`;
7. run `python scripts/validate_book.py`;
8. run `python scripts/next_book_action.py` to confirm the resulting state.

If validation fails, do not force the build or rewrite approval history. Repair unambiguous metadata/build drift; escalate any material chapter or approval conflict.

## Cross-chapter review

After every three to five integrated chapters, review the group as a learning path:

1. choose canonical homes for repeated explanations;
2. standardize terms in `glossary.md`;
3. verify prerequisites and clearly labeled previews;
4. add or repair relative cross-links;
5. check recurring examples for contradictory rules;
6. update manifest dependencies, priorities, and status projection;
7. record proposed structural changes before renumbering or moving content.

Prefer cross-references over copied explanations. Structural changes affecting approved chapters require human approval.

## Existing-work migration

Do not restart valid work. Infer evidence from existing artifacts, adopt the most advanced justified state, and record any migration exception explicitly. Chapter 02 is adopted as an approved-but-unintegrated baseline because its chapter, review, former status record, final rubric, and focused approval record agreed before the manifest existed. `legacy_status_consensus` is a migration-only evidence kind and must never be created for future approvals.
